# 3. Methodology

This study adopts a standardized benchmarking approach to evaluate the security posture of web application code generated by AI coding systems. The emphasis is on the default security behavior of each system when given a normal developer-style prompt without iterative correction.

## 3.1 Model Selection
The evaluation covers four contemporary AI coding systems:

| Evaluated System | Access Interface |
|------------------|------------------|
| Gemini 3.5 Flash | Antigravity IDE |
| DeepSeek V4 Flash | OpenCode via API |
| Kimi K2.5 | OpenCode via API |
| GPT-5.4-mini | OpenCode via API |

The paper treats these as evaluated systems rather than permanent model rankings. Model behavior can change over time due to provider updates, system prompts, and deployment settings, so provider-side changes remain a threat to reproducibility.

This setup reflects practical developer-facing workflows rather than isolated raw model API calls alone. Gemini was evaluated through Antigravity IDE, while the other three systems were evaluated through OpenCode connected to their respective APIs. Because coding-agent clients and IDEs may add system prompts, repository context, tool-use behavior, or session memory, the results should be interpreted as outputs from each model-interface configuration.

## 3.2 Evaluation Criteria: The "One-Shot" Constraint
To simulate a common developer interaction pattern, this study uses **one-shot prompting**. Each model is evaluated using its first generated output without follow-up instructions, corrections, or security-specific refinement. This design highlights the model's default security behavior, but it does not measure how secure the output could become after expert prompting or iterative repair.

## 3.3 Prompt Set

The benchmark consists of 10 prompts covering common web application functions: authentication, password reset, CRUD, SQL search, file upload, access control, payment secrets, contact form validation, frontend login, and session management. The same prompts were used across all evaluated systems to support comparability.

## 3.4 Analysis Protocol

The generated code was saved in model-specific and prompt-specific folders. The analysis process followed these steps:
1. Preserve the first generated output for each prompt and model.
2. Install dependencies only when required for functional review.
3. Run Semgrep OSS [R11] using the `auto` configuration against the generated source code.
4. Exclude dependency folders such as `node_modules` from analysis and public reporting.
5. Manually inspect selected findings and logic-sensitive areas, including authentication, session handling, file upload paths, and frontend exposure of sensitive values.
6. Map validated security findings to OWASP Top 10 categories.
7. Normalize duplicate tool reports so that repeated reports of the same root vulnerability are not overcounted.

## 3.5 Scope of Security Testing

The primary results in this paper are based on Static Application Security Testing (SAST) and manual review. Dynamic Application Security Testing (DAST) with OWASP ZAP is relevant future work, but it is not used as the basis for the aggregate results unless applied consistently to all generated samples. This distinction avoids mixing static findings from some samples with dynamic findings from others.

Semgrep's `--config auto` option selects registry rules based on the detected languages and frameworks in each generated project. These rules substantially overlap with OWASP-style web security weaknesses, but the raw output is not treated as a pure OWASP-only benchmark. Instead, Semgrep findings were manually reviewed, security-relevant findings were retained, duplicates were consolidated, and the validated findings were mapped to OWASP Top 10 categories for comparative reporting.

## 3.6 Prompt Rationale and Descriptions

| # | Prompt Category | Rationale |
|---|-----------------|-----------|
| 1 | Auth (JWT) | Tests handling of sensitive tokens and secure session management. |
| 2 | Auth (Reset) | Tests logic for one-time tokens and email security. |
| 3 | Database (CRUD) | Evaluates input validation and use of secure sub-resources (SRI). |
| 4 | Database (SQL) | Specifically targets SQL Injection (SQLi) vulnerabilities. |
| 5 | File Upload | Identifies Path Traversal and Local File Inclusion (LFI) risks. |
| 6 | Access Control | Tests complex logic for Role-Based Access Control (RBAC). |
| 7 | Secrets | Ensures models don't hardcode API keys or credentials. |
| 8 | Sanitization | Evaluates defense against Cross-Site Scripting (XSS) at the input level. |
| 9 | Frontend XSS | Tests the model's reliance on frontend frameworks' auto-escaping. |
| 10| Session Management| Evaluates secure cookie flags and session fixation defenses. |

## 3.7 Tooling
- **Semgrep OSS**: Primary tool for automated vulnerability identification.
- **Manual Code Review**: Used to identify logical flaws (e.g., hardcoded hashes) missed by automated rules.
- **Local Dev Environment**: Node.js, Python, and Docker environments for functional verification.

## 3.8 Counting and Severity Rules

A finding is counted as one unique vulnerability instance within a model-prompt sample after duplicate consolidation. Severity is based primarily on the scanner's reported severity. Manual findings are assigned severity based on exploitability and potential impact. For example, a hardcoded password hash exposed in frontend code is treated as a high-severity design flaw because it violates client-server trust boundaries and may enable credential analysis or misuse.
